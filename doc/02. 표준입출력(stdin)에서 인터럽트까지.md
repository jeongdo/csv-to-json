# 표준입출력(stdin)에서 인터럽트까지

## 시작점

Go를 공부하다 보면 자연스럽게 이런 코드를 만난다.

```go
var input string
fmt.Scanln(&input)
```

처음에는 단순히

> "키보드로 입력받는다"

정도로 이해한다.

하지만 여기서 의문이 생긴다.

- 키보드는 어떻게 프로그램까지 도달하는가?
- 왜 Enter를 눌러야 입력이 완료되는가?
- stdin은 누가 만드는가?

---

# 1. 표준입출력의 존재

운영체제는 프로세스를 시작할 때 기본적으로 세 가지 입출력 채널을 제공한다.

```text
stdin   (0) : 표준 입력
stdout  (1) : 표준 출력
stderr  (2) : 표준 에러
```

프로세스는 이 채널을 통해 외부와 통신한다.

```text
Go 프로그램
    │
    ├── stdin
    ├── stdout
    └── stderr
```

Go의 `fmt.Scanln()`은 stdin을 읽는다.

---

# 2. stdin은 키보드인가?

정확히 말하면 아니다.

stdin은 추상화된 입력 채널이다.

일반적인 터미널 환경에서는 다음과 같이 연결된다.

```text
키보드
 ↓
터미널
 ↓
stdin
 ↓
프로그램
```

따라서 프로그램은 키보드를 직접 알지 못한다.

오직 stdin만 알 뿐이다.

---

# 3. Enter를 눌러야 하는 이유

터미널은 보통 Line Buffering 모드로 동작한다.

사용자가 입력하면

```text
A
B
C
D
```

문자들이 터미널 입력 버퍼에 쌓인다.

```text
입력 버퍼

[A][B][C][D]
```

프로그램은 아직 받지 못한다.

Enter를 누르면

```text
[A][B][C][D][\n]
```

이 되고

터미널은

```text
ABCD\n
```

을 stdin으로 전달한다.

그 순간

```go
fmt.Scanln()
```

이 반환된다.

---

# 4. 브라우저 입력창은 왜 다르게 동작할까?

ChatGPT 입력창은 stdin을 사용하지 않는다.

브라우저는 이벤트 기반이다.

```text
키보드
 ↓
OS
 ↓
브라우저
 ↓
keydown 이벤트
 ↓
textarea 값 변경
 ↓
화면 갱신
```

그래서

```text
A
```

를 누르면 즉시 화면에 보인다.

Enter를 기다리지 않는다.

---

# 5. 이벤트라는 개념

GUI 프로그램은 입력 종류가 매우 많다.

```text
키보드
마우스
창 이동
창 크기 변경
네트워크 수신
타이머
```

이것을 전부 stdin으로 처리하기 어렵다.

그래서 등장한 것이 이벤트 모델이다.

```text
사건(Event) 발생
 ↓
이벤트 큐
 ↓
핸들러 호출
```

예)

```text
마우스 클릭
 ↓
MouseClick Event
 ↓
onclick()
```

---

# 6. 그런데 이벤트는 어디서 오는가?

여기서 MCU 경험이 연결된다.

ATmega128을 예로 들면

```text
버튼
 ↓
INT0 인터럽트
 ↓
ISR 실행
```

이 구조를 사용한다.

PC도 본질적으로 같다.

---

# 7. 키보드는 인터럽트를 사용한다

사용자가

```text
A
```

를 누른다.

그러면

```text
키보드
 ↓
하드웨어 인터럽트 발생
 ↓
CPU
 ↓
커널
```

이 된다.

중요한 점은

인터럽트 자체가 문자 데이터는 아니라는 것이다.

인터럽트는

```text
"키보드에 새 데이터가 있음"
```

이라는 알림이다.

커널은 이후 키보드 컨트롤러에서 실제 데이터를 읽는다.

---

# 8. 인터럽트 → 이벤트 → stdin

지금까지의 내용을 모두 연결하면

```text
키보드 하드웨어
 ↓
하드웨어 인터럽트
 ↓
커널 드라이버
 ↓
터미널 입력 버퍼
 ↓
stdin
 ↓
Go fmt.Scanln()
```

이 된다.

이 한 줄이 표준입출력의 본질이다.

---

# 9. MCU 경험과 현대 소프트웨어의 연결

MCU 시절에는

```text
버튼
 ↓
인터럽트
 ↓
ISR
```

만 보였다.

Java 개발 시절에는

```text
System.in
```

만 보였다.

Go/Rust를 다시 공부하면서 비로소

```text
인터럽트
 ↓
드라이버
 ↓
버퍼
 ↓
stdin
 ↓
언어 런타임
```

전체 계층이 연결되기 시작한다.

---

# 최종 그림

CLI 프로그램

```text
키보드
 ↓
하드웨어 인터럽트
 ↓
커널 드라이버
 ↓
터미널 입력 버퍼
 ↓
stdin
 ↓
Go fmt.Scanln()
```

GUI 프로그램

```text
키보드
 ↓
하드웨어 인터럽트
 ↓
커널 드라이버
 ↓
OS 이벤트 큐
 ↓
브라우저 이벤트
 ↓
keydown
 ↓
JavaScript
```

둘은 완전히 다른 것이 아니다.

둘 다 같은 출발점에서 시작한다.

```text
하드웨어 인터럽트
```

CLI는 그것을 "문자열 입력(stdin)"으로 추상화한 것이고,

GUI는 그것을 "이벤트(Event)"로 추상화한 것이다.
