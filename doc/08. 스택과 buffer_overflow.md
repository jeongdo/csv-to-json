# 8. 버퍼 오버플로우(Buffer Overflow) 완전 정리

## 1. 스택 메모리 기본 구조

스택이 높은 주소 → 낮은 주소 방향으로 자라는 건 **CPU/OS 하드웨어 기본 스펙**이며, 모든 언어가 동일하게 사용한다.

```
낮은 주소 0x0000
┌──────────────┐
│     코드     │  ← 프로그램 명령어
│──────────────│
│     데이터   │  ← 전역변수
│──────────────│
│      힙      │  ↓ malloc 할때마다 아래로 증가
│              │
│   (빈공간)   │  ← 힙과 스택이 빈공간을 공유
│              │
│      스택    │  ↑ 함수 호출할때마다 위로 증가
└──────────────┘
높은 주소 0xFFFF
```

> 힙과 스택이 서로 마주보며 자라서 빈공간을 효율적으로 공유하는 설계.

---

## 2. 함수 호출과 Return Address

### 코드 영역에서의 흐름

```
main() 코드 영역
┌─────────────────────────┐
│  ...                    │
│  0x1000: login(input)   │  ← 여기서 호출
│  0x1004: printf("ok")   │  ← 호출 끝나면 여기로 돌아와야 함
│  ...                    │
└─────────────────────────┘
```

### login() 호출 순간 CPU가 하는 일

```
"나 지금 login으로 점프하는데,
 끝나면 0x1004로 돌아와야 하니까
 그 주소를 스택에 메모해놓자"

→ 스택에 0x1004 PUSH
→ login() 으로 점프
```

### 결과적으로 스택에 쌓이는 구조

```
높은 주소
┌─────────────────┐
│  Return Address │  ← "login 끝나면 0x1004로 돌아가" (CPU가 자동 저장)
│─────────────────│
│  SFP (이전 BP)  │  ← 이전 스택 프레임 포인터
│─────────────────│
│  buf[63]        │
│  buf[62]        │
│  ...            │
│  buf[1]         │
│  buf[0]         │  ← strcpy는 여기서부터 채우기 시작
└─────────────────┘
낮은 주소
```

> **Return Address** = 코드 영역의 주소값을 스택에 복사해둔 것.  
> 해커가 이 숫자를 악성코드 주소로 바꿔치기하면, 함수 끝날 때 CPU가 악성코드로 점프해버린다.

---

## 3. 왜 C만 취약한가

### C의 배열 = 그냥 메모리 주소

```c
void login(char *input) {
    char buf[64];
    strcpy(buf, input);  // 크기 검사 없음 → 넘치면 그냥 씀
}
```

`buf[100]`은 C에서 `*(buf + 100)` 과 동일.  
"buf 시작주소에서 100 더한 곳에 써라" — 거기가 Return Address든 뭐든 모른다.

### 언어별 비교

| 언어 | 배열 경계 검사 | 버퍼 오버플로우 가능? |
|------|--------------|-------------------|
| C / C++ | ❌ 없음 | ✅ 가능 |
| Java | ✅ 런타임 검사 | ❌ 불가 |
| Python | ✅ 런타임 검사 | ❌ 불가 |
| Go | ✅ 런타임 검사 | ❌ 불가 |
| Rust | ✅ 컴파일타임 검사 | ❌ 불가 |

> 스택 구조는 모든 언어가 동일. C만 그 메모리에 무제한 접근이 가능한 것.

---

## 4. 버퍼 오버플로우 공격 원리

### 오버플로우 발생 시 메모리

```
buf[0] buf[1] ... buf[63] | SFP | Return Address
  ↑ 여기서 시작해서 오른쪽으로 계속 씀 →→→→→→→→→
```

- buf[64] 이후 → SFP 영역 침범
- buf[68] 이후 → **Return Address 덮어써짐**

### 해커의 정밀 입력 구성

Return Address까지 거리 = buf(64) + SFP(4) = **68바이트**

```
입력값 구성:
[쓰레기 68바이트] + [악성코드 주소 4바이트]
     ↑                      ↑
  그냥 채우기용          Return Address 덮어씀
```

---

## 5. 현대의 방어막

### ① Stack Canary (컴파일러 수준)

```
[ buf ] [ canary값 ] [ SFP ] [ RET ]
```

- buf 넘치면 canary가 먼저 오염됨
- 함수 리턴 전 canary 값 검사 → 변조 시 즉시 프로세스 종료
- GCC `-fstack-protector` 기본 활성화

### ② ASLR (OS 수준)

- 스택, 힙, 라이브러리 주소를 **실행할 때마다 랜덤 배치**
- 악성코드 주소를 하드코딩해도 매번 틀림

### ③ NX / DEP (하드웨어 수준)

- 스택 메모리를 **실행 불가** 영역으로 표시
- 셸코드를 스택에 심어도 CPU가 실행 자체를 거부

---

## 6. 현대 해커의 우회 기법

| 기법 | 원리 |
|------|------|
| **ROP** (Return Oriented Programming) | 셸코드 안 심음. 이미 실행 가능한 코드 조각(가젯)들을 체인처럼 연결 |
| **Heap Overflow** | 스택 대신 힙 오버플로우로 우회 |
| **Format String** | `printf(buf)` 같은 실수로 메모리 읽기/쓰기 |
| **Use-After-Free** | 해제된 메모리 재사용 |

## 7. 실제 공격 과정 (Fuzzing)

해커가 오프셋을 찾는 실제 과정:

```
1단계: 탐색
입력 100바이트 → 500 에러 (터짐)
입력 50바이트  → 정상응답  (아직 안넘침)
→ "buf가 50~100 사이 어딘가구나"

2단계: 정밀화
입력 68바이트째부터 응답이 이상하게 바뀜
→ "여기가 Return Address 시작이구나"

3단계: 익스플로잇
[쓰레기 68바이트] + [악성코드 주소] 전송
→ 함수 리턴 시 악성코드로 점프
```

이걸 자동화한 도구들:

```python
# fuzzer 개념 코드
for i in range(1, 500):
    send(b"A" * i)
    if response == "crash":
        print(f"버퍼 크기는 {i-1} 근처")
        break
```

- **Metasploit** - 자동으로 패턴 생성, 오프셋 계산
- **pwntools** - 파이썬으로 익스플로잇 자동화
- **fuzzer** - 무작위 입력을 고속으로 반복 전송

## 8. 최종 요약

| 항목 | 내용 |
|------|------|
| 스택 구조 | CPU/OS 기본 스펙. 모든 언어 동일 |
| Return Address | 함수 호출 시 CPU가 스택에 자동 저장하는 복귀 주소 |
| C의 취약점 | 배열 경계 검사 없음 → 옆 메모리 무제한 접근 가능 |
| 공격 원리 | buf 오버플로우로 Return Address를 악성코드 주소로 덮어씀 |
| 현대 방어 | Canary + ASLR + NX로 난이도 급상승 |
| 공격 진화 | ROP 등 방어막 우회 기법으로 대응 |
| 탐색 방법 | Fuzzing으로 오프셋 자동 탐색 |

> **결론**: 버퍼 오버플로우는 "C가 메모리를 검사 없이 직접 다룬다"는 특성과 "스택에 Return Address가 지역변수 바로 옆에 있다"는 구조가 만나서 생기는 취약점. 방어막 덕에 단순 공격은 막혔지만, 우회 기법과 함께 지금도 현역 취약점 카테고리.
