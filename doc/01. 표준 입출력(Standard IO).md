# 표준 입출력(Standard I/O)과 파일 디스크립터(File Descriptor)

## 1. 표준 입출력이란?

표준 입출력(Standard I/O)은 운영체제가 모든 프로세스에게 기본적으로 제공하는 3개의 입출력 채널이다.

프로세스가 생성되면 운영체제는 다음 3개의 파일 디스크립터(FD)를 자동으로 연결한다.

| FD | 이름     | 역할       |
| -- | ------ | -------- |
| 0  | stdin  | 표준 입력    |
| 1  | stdout | 표준 출력    |
| 2  | stderr | 표준 에러 출력 |

```text
FD 0 → stdin
FD 1 → stdout
FD 2 → stderr
```

즉 프로그램이 실행되면 운영체제가

> "입력용 하나, 출력용 하나, 에러용 하나를 줄 테니 알아서 사용해라"

라고 기본 통로를 제공하는 개념이다.

---

## 2. 프로그램과 프로세스

프로그램은 디스크에 저장된 실행 파일이다.

```text
java.exe
python.exe
myapp
```

실행되면 운영체제가 프로세스를 생성한다.

```text
프로그램
    ↓ 실행
프로세스 생성
    ↓
OS가 FD 0,1,2 연결
```

따라서 표준 입출력은 언어 기능이 아니라 운영체제 기능이다.

---

## 3. 언어들은 단순히 래핑만 한다

### Java

```java
System.in
System.out
System.err
```

### Python

```python
sys.stdin
sys.stdout
sys.stderr
```

### Go

```go
os.Stdin
os.Stdout
os.Stderr
```

언어들은 운영체제가 제공한 FD 0,1,2를 사용하기 쉽게 감싸서 제공할 뿐이다.

---

## 4. 기본 연결 구조

터미널에서 프로그램을 실행하면 보통 다음과 같이 연결된다.

```text
키보드
   ↓
stdin (0)

프로그램

stdout (1) → 터미널 화면
stderr (2) → 터미널 화면
```

따라서

```java
System.out.println("Hello");
```

는 stdout(FD 1)으로 출력된다.

---

## 5. stdout 과 stderr 를 분리한 이유

정상 출력과 에러 출력을 분리하기 위해서이다.

예시:

```bash
java Main > output.log
```

실행 후:

```text
stdout → output.log
stderr → 터미널
```

즉 정상 메시지만 파일로 저장된다.

에러만 저장하려면:

```bash
java Main 2> error.log
```

결과:

```text
stdout → 터미널
stderr → error.log
```

둘 다 저장하려면:

```bash
java Main > all.log 2>&1
```

---

## 6. 프로세스는 출력 위치를 모른다

예를 들어 Java 코드:

```java
System.out.println("Hello");
```

프로그램 입장에서는 단순히 stdout(FD 1)에 출력했을 뿐이다.

원래는:

```text
stdout(FD1)
    ↓
터미널
```

이지만

```bash
java Main > result.txt
```

로 실행하면 운영체제가 미리 연결을 변경한다.

```text
stdout(FD1)
    ↓
result.txt
```

따라서 프로그램은 수정 없이도 파일 출력이 가능하다.

---

## 7. 파일 디스크립터(File Descriptor)

리눅스는 모든 입출력을 파일 디스크립터(FD)로 관리한다.

표준 입출력도 그 일부일 뿐이다.

```text
0 = stdin
1 = stdout
2 = stderr
```

파일을 열면 새로운 FD가 생성된다.

예시:

```c
int fd = open("debug.log", O_WRONLY);
```

결과:

```text
0 = stdin
1 = stdout
2 = stderr
3 = debug.log
```

또 다른 파일을 열면:

```text
4 = another.log
```

처럼 증가한다.

---

## 8. Bash에서 FD 생성

새로운 FD를 직접 만들 수도 있다.

```bash
exec 3> debug.log
```

결과:

```text
0 = stdin
1 = stdout
2 = stderr
3 = debug.log
```

쓰기:

```bash
echo "debug message" >&3
```

---

## 9. Java 예외와 stderr

Java에서 처리되지 않은 예외가 발생하면 기본적으로 stderr(FD 2)에 출력된다.

예제:

```java
public static void main(String[] args) {
    throw new RuntimeException("에러 발생");
}
```

출력:

```text
Exception in thread "main" java.lang.RuntimeException: 에러 발생
    ...
```

이 메시지는 stdout이 아니라 stderr로 출력된다.

따라서:

```bash
java Main > output.log
```

를 실행해도 예외 메시지는 터미널에 계속 보인다.

이유:

```text
stdout → output.log
stderr → 터미널
```

---

## 10. System.out 과 System.err

Java에는 두 개의 출력 스트림이 있다.

```java
System.out.println("정상 메시지");
System.err.println("에러 메시지");
```

내부적으로는:

```text
System.out → stdout(FD1)
System.err → stderr(FD2)
```

와 연결되어 있다.

---

## 11. printStackTrace()

다음 코드는:

```java
catch (Exception e) {
    e.printStackTrace();
}
```

기본적으로 stderr에 출력한다.

개념적으로는:

```java
catch (Exception e) {
    System.err.println(...);
}
```

와 비슷하게 동작한다.

따라서 콘솔에 보이는 대부분의 스택 트레이스는 FD 2(stderr)를 통해 출력된다.

---

## 12. 리눅스의 핵심 철학

리눅스는 다음을 모두 동일한 방식으로 다룬다.

* 파일
* 터미널
* 소켓
* 파이프

예시:

```text
0 = stdin
1 = stdout
2 = stderr
3 = 파일
4 = 파일
5 = 소켓
6 = 파이프
```

즉 리눅스에서는 모든 입출력이 파일 디스크립터라는 동일한 인터페이스로 관리된다.

## 핵심 요약

프로세스가 생성되면 운영체제는 기본적으로 다음 3개의 파일 디스크립터를 제공한다.

```text
0 = stdin
1 = stdout
2 = stderr
```

언어들은 이를 래핑해서 제공할 뿐이다.

```java
System.in   → FD 0
System.out  → FD 1
System.err  → FD 2
```

또한 Java의 예외 메시지와 스택 트레이스는 기본적으로 stderr(FD 2)를 통해 출력된다.

## 13. 표준 입출력은 공유되는가?

처음에는 다음과 같은 오해를 하기 쉽다.

> 프로세스마다 stdin, stdout, stderr를 준다고 했으니 실제 출력 객체도 프로세스마다 새로 생성되는 것 아닌가?

실제로는 그렇지 않다.

프로세스마다 생성되는 것은 FD(File Descriptor) 테이블이다.

```text
프로세스 A
 FD 0
 FD 1
 FD 2

프로세스 B
 FD 0
 FD 1
 FD 2
```

즉 FD 자체는 프로세스마다 별도로 존재한다.

---

## 14. 실제 출력 대상은 공유될 수 있다

예를 들어 같은 터미널에서 두 개의 프로그램을 실행했다고 가정하자.

```bash
java App1
java App2
```

개념적으로는 다음과 같다.

```text
프로세스 A FD1 ─┐
                │
                ▼
             터미널

프로세스 B FD1 ─┘
```

즉

```text
FD 테이블 = 프로세스별
실제 출력 대상 = 공유 가능
```

이다.

---

## 15. FD는 일종의 참조(Reference)

자바 객체 참조를 생각하면 이해하기 쉽다.

```java
Terminal t1 = sharedTerminal;
Terminal t2 = sharedTerminal;
```

두 변수는 서로 다르지만 같은 객체를 가리킨다.

FD도 비슷하다.

```text
프로세스 A
 FD1 -> 터미널

프로세스 B
 FD1 -> 터미널
```

FD는 각 프로세스에 존재하지만 실제 대상은 동일할 수 있다.

---

## 16. 리눅스 커널의 개념적 구조

커널 내부는 개념적으로 다음과 같은 계층으로 관리된다.

```text
Process
   ↓
FD Table
   ↓
Open File Object
   ↓
Terminal / File / Socket / Pipe
```

예시:

```text
프로세스 A
   FD1
     │
     ▼
OpenFile#123
     │
     ▼
   터미널

프로세스 B
   FD1
     │
     ▼
OpenFile#123
     │
     ▼
   터미널
```

여러 프로세스가 동일한 출력 대상을 공유할 수 있다.

---

## 17. 핵심 정리

표준 입출력은 프로세스마다 독립적으로 존재한다.

```text
프로세스 A
 0 = stdin
 1 = stdout
 2 = stderr

프로세스 B
 0 = stdin
 1 = stdout
 2 = stderr
```

하지만 실제 출력 대상은 공유될 수 있다.

```text
프로세스 A FD1 ─┐
                │
                ▼
             터미널

프로세스 B FD1 ─┘
```

따라서

- FD 테이블은 프로세스마다 생성된다.
- 터미널, 파일, 소켓, 파이프 같은 실제 자원은 공유될 수 있다.
- FD는 실제 자원을 가리키는 참조(Reference)에 가깝다.

이것이 리눅스 입출력 모델의 핵심 개념 중 하나이다.
