# 프로그램은 커널을 어떻게 부르는가? - 시스템콜

## 시작점

이전 글에서 커널의 존재를 이해했다.

```text
프로그램
 ↓
커널
 ↓
하드웨어
```

그런데 새로운 의문이 생긴다.

> 프로그램은 커널에게 어떻게 요청하는가?

답은 시스템콜(System Call)이다.

---

# 1. 프로그램은 하드웨어를 직접 건드릴 수 없다

MCU에서는

```c
PORTA = 0xFF;
```

같은 코드가 가능했다.

하지만 Linux에서는 불가능하다.

```text
응용프로그램
 X
 하드웨어 직접 접근 금지
```

대신

```text
응용프로그램
 ↓
시스템콜
 ↓
커널
 ↓
하드웨어
```

를 사용한다.

---

# 2. 시스템콜은 커널의 출입문

시스템콜은

> 프로그램이 커널에게 기능을 요청하는 공식 창구

이다.

예)

```text
파일 읽기
파일 쓰기
프로세스 생성
소켓 생성
메모리 할당
```

모두 시스템콜을 통해 수행된다.

---

# 3. fmt.Scanln()의 진짜 동작

겉으로 보면

```go
fmt.Scanln()
```

만 보인다.

하지만 내부에서는

```text
fmt.Scanln()
 ↓
Go 라이브러리
 ↓
read() 시스템콜
 ↓
커널
 ↓
stdin 버퍼
```

가 발생한다.

중요한 점은

```text
fmt.Scanln()
```

은 시스템콜이 아니다.

```text
read()
```

가 시스템콜이다.

---

# 4. 시스템콜과 인터럽트

지금까지 두 종류의 호출을 만났다.

## 하드웨어가 커널을 부르는 경우

```text
키보드
 ↓
하드웨어 인터럽트
 ↓
커널
```

## 프로그램이 커널을 부르는 경우

```text
프로그램
 ↓
시스템콜
 ↓
커널
```

둘 다 결국 커널로 진입한다.

---

# 5. 중요한 시스템콜들

대표적인 예

```text
read()
write()
open()
close()

socket()

fork()
execve()

mmap()
```

우리가 사용하는 대부분의 API는

결국 이 시스템콜들 위에 만들어져 있다.

---

# 6. Java도 결국 시스템콜이다

```java
System.out.println("Hello");
```

실제로는

```text
JVM
 ↓
write() 시스템콜
 ↓
커널
```

이 된다.

---

# 7. Go도 결국 시스템콜이다

```go
fmt.Println("Hello")
```

실제로는

```text
fmt.Println()
 ↓
Go Runtime
 ↓
write() 시스템콜
 ↓
커널
```

이 된다.

---

# 최종 그림

하드웨어가 요청할 때

```text
인터럽트
 ↓
커널
```

프로그램이 요청할 때

```text
시스템콜
 ↓
커널
```

즉,

```text
인터럽트 = 하드웨어가 커널을 부르는 문

시스템콜 = 프로그램이 커널을 부르는 문
```

이다.

운영체제를 이해한다는 것은 결국

커널과 시스템콜의 관계를 이해하는 것이다.
