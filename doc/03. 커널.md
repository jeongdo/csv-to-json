# 인터럽트 뒤에는 누가 있는가? - 커널의 등장

## 시작점

이전 글에서는 다음 흐름을 이해했다.

```text
키보드
 ↓
하드웨어 인터럽트
 ↓
커널 드라이버
 ↓
터미널 입력 버퍼
 ↓
stdin
 ↓
fmt.Scanln()
```

여기서 새로운 의문이 생긴다.

> 인터럽트는 누가 처리하는가?

> 드라이버는 누가 관리하는가?

> 수많은 하드웨어는 누가 통제하는가?

이 질문의 답이 바로 커널(Kernel)이다.

---

# 1. 커널은 운영체제의 핵심부

많은 사람이

```text
운영체제 = Windows
운영체제 = Linux
```

정도로 생각한다.

하지만 내부를 보면

```text
운영체제
 ├─ 커널
 ├─ 라이브러리
 ├─ 쉘
 ├─ 유틸리티
```

구조를 가진다.

커널은 그 중심에 있는 핵심 소프트웨어다.

---

# 2. MCU에서는 내가 커널이었다

ATmega128 시절을 생각해 보자.

```text
버튼
 ↓
인터럽트
 ↓
ISR
```

ISR 안에서

```c
PORTA = 0xFF;
```

같은 코드를 작성했다.

즉,

```text
인터럽트 처리
장치 제어
메모리 사용
```

을 모두 개발자가 직접 했다.

사실상 개발자가 커널 역할을 한 셈이다.

---

# 3. PC에서는 커널이 대신한다

현대 운영체제에서는 응용프로그램이 직접 하드웨어를 건드릴 수 없다.

```text
프로그램
 ↓
커널
 ↓
하드웨어
```

구조를 가진다.

예)

```go
fmt.Println("Hello")
```

실행

```text
프로그램
 ↓
커널
 ↓
터미널 드라이버
 ↓
화면 출력
```

---

# 4. 드라이버와 커널

커널은 모든 하드웨어를 직접 알 수 없다.

그래서 제조사는 장치별 드라이버를 제공한다.

예)

```text
키보드 드라이버
마우스 드라이버
SSD 드라이버
랜카드 드라이버
GPU 드라이버
```

커널은 이 드라이버들을 관리한다.

```text
프로그램
 ↓
커널
 ↓
드라이버
 ↓
하드웨어
```

---

# 5. 왜 드라이버 규칙이 필요한가

세상에는 수많은 제조사가 존재한다.

```text
Intel
AMD
NVIDIA
Samsung
Realtek
...
```

각자 자기 방식으로 만들면 운영체제는 유지될 수 없다.

그래서 커널은

```text
이런 함수로 등록하라
이런 인터럽트 방식을 사용하라
이런 메모리 관리 규칙을 따르라
```

라는 공통 규칙을 제공한다.

드라이버는 이 규칙에 맞게 작성되어야 한다.

---

# 6. 리눅스 커널과 리누스 토발즈

리눅스 커널은 전 세계 개발자들이 함께 개발하는 거대한 프로젝트다.

주요 관심사는

```text
프로세스 관리
메모리 관리
파일 시스템
인터럽트 처리
드라이버 프레임워크
```

같은 운영체제 핵심부다.

---

# 최종 그림

```text
응용프로그램
 ↓
커널
 ↓
드라이버
 ↓
하드웨어
```

MCU 시절

```text
나
 ↓
하드웨어
```

였다면

현대 운영체제에서는

```text
나
 ↓
커널
 ↓
하드웨어
```

가 된다.

커널은 하드웨어 세계와 소프트웨어 세계 사이의 관리자다.
